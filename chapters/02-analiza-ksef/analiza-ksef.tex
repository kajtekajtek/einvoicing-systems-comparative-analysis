\section{Analiza Krajowego Systemu e-Faktur}
\label{sec:ksef}

\subsection{Kontekst i podstawy prawne}
\label{sec:ksef-kontekst}

\textbf{Krajowy System e-Faktur (KSeF)} to państwowy system teleinformatyczny obowiązujący w Polsce, służący do centralnej wymiany i archiwizacji faktur ustrukturyzowanych zgodnych ze strukturą logiczną FA(3). Stanowi on realizację powszechnego obowiązku fakturowania elektronicznego, który jest sukcesywnie wdrażany dla poszczególnych grup przedsiębiorców począwszy od lutego 2026~r. \cite{ksef-w-pigulce,ksef-plan-wdrozenia}

\subsection{Architektura systemu}
\label{sec:ksef-architektura}

KSeF został zaprojektowany w modelu \textbf{scentralizowanego repozytorium}. Wszystkie e-faktury wystawiane w obrocie krajowym B2B (\textit{Business to Business}) trafiają do jednego systemu prowadzonego przez państwo, który pełni jednocześnie rolę kanału przekazu dokumentu oraz jego archiwizatora przez okres 10 lat. \cite{ksef-w-pigulce}

\subsubsection{Komponenty logiczne}
\label{sec:ksef-komponenty}

Z perspektywy integratora system składa się z trzech współpracujących komponentów: \cite{ksef-docs-repo}

\begin{itemize}
    \item \textbf{System wywołujący} -- oprogramowanie podatnika inicjujące komunikację z KSeF, np. system finansowo-księgowy, ERP, aplikacja mobilna.
    \item \textbf{REST API KSeF} -- publiczny interfejs programistyczny udostępniający operacje wystawiania, odbierania, wyszukiwania faktur oraz zarządzania uprawnieniami i certyfikatami. Specyfikacja interfejsu stanowi jedyny kontrakt integracyjny pomiędzy systemami zewnętrznymi a KSeF. \cite{ksef-publikacja-api}
    \item \textbf{System KSeF} -- centralne repozytorium faktur, moduł uprawnień, moduł certyfikatów wewnętrznych oraz mechanizm wystawiania Urzędowych Poświadczeń Odbioru (UPO). \cite{ksef-docs-przeglad}
\end{itemize}

\subsubsection{Kanały dostępu}
\label{sec:ksef-kanaly-dostepu}

System umożliwia kilka równoległych kanałów dostępu: \cite{ksef-plan-wdrozenia}

\begin{itemize}
    \item \textbf{REST API} -- podstawowy kanał dla integratorów i systemów ERP, opisany w sekcji \ref{sec:ksef-model-komunikacji}.
    \item \textbf{Aplikacja Podatnika KSeF} -- bezpłatna aplikacja webowa, przeznaczona dla podatników niekorzystających z własnego oprogramowania.
    \item \textbf{Aplikacja Mobilna KSeF} -- aplikacja na systemy Android oraz iOS, umożliwiająca wystawianie, przeglądanie i weryfikację faktur z poziomu urządzeń mobilnych.
\end{itemize}

\subsubsection{Środowiska systemu}
\label{sec:ksef-srodowiska}

System dostępny jest w trzech niezależnych środowiskach, różniących się danymi, limitami wywołań API (\textit{rate limits}) oraz skutkami prawnymi wystawianych dokumentów:

\begin{itemize}
    \item \textbf{Środowisko testowe (integracyjne)} -- przeznaczone do testów integracyjnych z danymi syntetycznymi. Dostępne pod adresem \texttt{api-test.ksef.mf.gov.pl}, zawiera dodatkowy interfejs programistyczny pozwalający zasymulować rejestrację podmiotu, nadanie uprawnień ZAW-FA czy utworzenie grupy VAT. \cite{ksef-docs-przeglad,ksef-start-testow}
    \item \textbf{Środowisko przedprodukcyjne (Demo)} -- służące do weryfikacji końcowej przed wdrożeniem produkcyjnym, pracuje na rzeczywistych danych uwierzytelniających podatników, lecz wystawione w nim faktury nie wywierają skutków prawnych. \cite{ksef-udostepnienie-przedprod}
    \item \textbf{Środowisko produkcyjne} -- środowisko docelowe, w którym wystawiane faktury wywierają pełne skutki prawne. \cite{ksef-udostepnienie-prod}
\end{itemize}

\subsubsection{Tryby pracy}
\label{sec:ksef-tryby-pracy}

Architektura KSeF przewiduje cztery tryby wystawiania faktur. Dwa z nich -- tryby online i~offline24 -- inicjowane są przez podatnika. Pozostałe dwa tryby uruchamiane są w~razie zakłóceń działania systemu. \cite{ksef-plan-wdrozenia,ksef-tryby-offline}

\begin{itemize}
    \item \textbf{Tryb online} -- faktura jest wystawiana bezpośrednio w KSeF, a numer KSeF (oraz datę wystawienia) nadaje system w momencie odbioru.
    \item \textbf{Tryb offline24} -- faktura wystawiana jest lokalnie u podatnika, a następnie przesyłana do KSeF nie później niż następnego dnia roboczego. Tryb ten wymaga posłużenia się \textbf{certyfikatem KSeF typu~2}, wydawanym przez Ministerstwo Finansów, wymaganym do oznaczenia faktury kodem potwierdzającym tożsamość wystawcy przy wystawianiu poza systemem. \cite{ksef-certyfikaty}
    \item \textbf{Tryb offline} -- uruchamiany w przypadku niedostępności KSeF. Faktura wystawiana jest poza systemem, a~podatnik ma obowiązek dostarczyć ją do KSeF najpóźniej następnego dnia roboczego po zakończeniu niedostępności.
    \item \textbf{Tryb awaryjny} -- aktywowany w przypadku awarii systemu. Termin na dosłanie faktury wynosi 7~dni roboczych od zakończenia awarii; kolejny komunikat o~awarii wydłuża ten termin. W~przypadku awarii całkowitej obowiązek przesyłania faktur do KSeF jest w~całości zniesiony, a~fakturę można wystawić w~dowolnej formie bez struktury FA(3).
\end{itemize}

\subsection{Wzór faktury -- struktura logiczna FA(3)}
\label{sec:ksef-wzor-faktury}

Schemat XSD (\textit{XML Schema Definition}) jest formalną specyfikacją opisującą strukturę, typy danych i reguły poprawności dokumentu XML. Każda faktura ustrukturyzowana przyjmowana przez KSeF musi być sporządzona w formacie XML zgodnym ze schematem XSD o nazwie \textbf{FA(3)}. \cite{ksef-w-pigulce}

\subsubsection{Poziomy walidacji dokumentu}
\label{sec:ksef-walidacja}

Każda faktura przechodzi przez cztery poziomy walidacji, których pozytywne ukończenie jest warunkiem przyjęcia dokumentu i wystawienia Urzędowego Poświadczenia Odbioru (UPO).

\begin{enumerate}
    \item \textbf{Walidacja syntaktyczna} weryfikuje poprawność dokumentu XML w kontekście specyfikacji \textit{XML W3C}~1.0 oraz wymagań systemu, a także zgodność z zadeklarowanym przy otwarciu sesji schematem XSD.

    \item \textbf{Walidacja techniczna} obejmuje weryfikację maksymalnego rozmiaru pliku oraz limitu ilości faktur w sesji. Na tym etapie sprawdzana jest również poprawność kryptograficzna: użycie algorytmu \texttt{AES-256-CBC} z dopełnieniem PKCS\#7 oraz zaszyfrowanie klucza symetrycznego algorytmem \texttt{RSAES-OAEP} (SHA-256/MGF1). W sesji interaktywnej dodatkowo wyliczane i porównywane są skróty kryptograficzne faktury jawnej oraz zaszyfrowanej wraz z ich rozmiarami.

    \item \textbf{Walidacja semantyczna (biznesowa)} weryfikuje poprawność logiczną i zgodność danych z regułami biznesowymi. Obejmuje m.in.: weryfikację sumy kontrolnej numerów NIP, walidację dat wystawienia faktury, a także weryfikację unikalności faktury, omówioną w sekcji~\ref{sec:ksef-fa3-duplikaty}.

    \item \textbf{Walidacja uprawnień i kontekstu} -- równolegle do walidacji treści system kontroluje, czy uwierzytelniony podmiot posiada odpowiednie uprawnienia do wystawienia faktury. Mechanizm ten korzysta z modułu uprawnień KSeF i obejmuje weryfikację ról nadanych zarówno bezpośrednio (np.\ właścicielowi NIP), jak i pośrednio (np.\ biuru rachunkowemu działającemu jako \texttt{PodmiotUpowazniony}).
\end{enumerate}

Każdy zidentyfikowany błąd zwracany jest klientowi w odpowiedzi API w postaci numerycznego kodu błędu wraz z opisem. Pełna lista kodów publikowana jest jako część specyfikacji OpenAPI. \cite{ksef-openapi,ksef-weryfikacja-faktury}

\subsubsection{Weryfikacja duplikatów i unikalność}
\label{sec:ksef-fa3-duplikaty}

KSeF weryfikuje unikalność faktur na podstawie kombinacji trzech pól: NIP sprzedawcy (\texttt{Podmiot1/NIP}), rodzaju faktury (\texttt{RodzajFaktury}) oraz numeru faktury (\texttt{P\_2}). Próba przesłania faktury o identycznej kombinacji tych pól skutkuje odrzuceniem z kodem błędu~440 (,,Duplikat faktury''). Unikalność jest weryfikowana przez 10 pełnych lat od końca roku kalendarzowego wystawienia faktury. \cite{ksef-weryfikacja-faktury}



\subsection{Model komunikacji}
\label{sec:ksef-model-komunikacji}

Komunikacja z Systemem KSeF realizowana jest poprzez jednolity interfejs programistyczny. Architektura rozwiązania wymaga stosowania standardów kryptograficznych zarówno na poziomie uwierzytelniania, jak i szyfrowania przesyłanych dokumentów. \cite{ksef-docs-przeglad}

\subsubsection{Interfejs programistyczny}
\label{sec:ksef-interfejs-programistyczny}

System udostępnia interfejs w architekturze \textbf{REST API}. Pełna specyfikacja kontraktu publikowana jest przez Ministerstwo Finansów w standardzie \texttt{OpenAPI 3.0.4}. 

Do komunikacji wykorzystywane są dwa formaty danych: \cite{ksef-weryfikacja-faktury}
\begin{itemize}
    \item \textbf{JSON} -- wykorzystywany do przesyłania metadanych, zapytań o status, zarządzania uprawnieniami oraz wymiany tokenów,
    \item \textbf{XML} -- używany wyłącznie do przesyłania samej treści faktur (w strukturze FA) oraz pobierania Urzędowego Poświadczenia Odbioru (UPO).
\end{itemize}

Interfejs wykorzystuje mechanizm \textit{rate limitingu}, w którym limity żądań określane są na sekundę, minutę i godzinę. Zakresy i wartości limitów są publicznie udostępniane, zróżnicowane w zależności od środowiska oraz dostosowane do charakteru operacji. \cite{ksef-docs-przeglad}

\subsubsection{Uwierzytelnianie i autoryzacja}
\label{sec:ksef-uwierzytelnianie-i-autoryzacja}

System wykorzystuje w pełni asynchroniczny mechanizm uwierzytelniania oparty o \textbf{tokeny JWT (JSON Web Token)}, rozdzielając ten proces od cyklu życia sesji wysyłkowej (\ref{sec:ksef-cykl-zycia-sesji}). Podstawą uzyskania dostępu jest powiązanie \textit{kontekstu logowania} (podmiotu, w imieniu którego wykonywane będą operacje w systemie) z \textit{podmiotem uwierzytelniającym} (podmiotem podejmującym próbę uwierzytelnienia). \cite{ksef-docs-przeglad}

Proces oparty jest na mechanizmie wyzwania (\textit{Challenge-Response}) i składa się z następujących kroków: \cite{ksef-docs-repo}

\begin{enumerate}
    \item \textbf{Wyzwanie (\textit{Auth Challenge}):} Klient za pomocą wywołania \texttt{POST /auth/challenge} pobiera jednorazowy znacznik ważny przez 10 minut.
    
    \item \textbf{Podpisanie żądania:} Klient wybiera jedną z dwóch ścieżek autoryzacji:
    \begin{itemize}
        \item \textbf{XAdES:} Klient buduje dokument XML (\texttt{AuthTokenRequest}) i podpisuje go podpisem w formacie XAdES, wykorzystując do tego certyfikat kwalifikowany, profil zaufany, certyfikat KSeF lub certyfikat dostawcy usług PEPPOL (\ref{sec:porownanie-4corner}). Dokument wysyłany jest asynchronicznie na \texttt{POST /auth/xades-signature}.
        \item \textbf{Token KSeF:} Wygenerowany wcześniej wewnętrzny token systemowy połączony ze znacznikiem czasu z wyzwania zostaje zaszyfrowany za pomocą algorytmu RSA-OAEP kluczem publicznym systemu i wysłany na \texttt{POST /auth/ksef-token}.
    \end{itemize}
    
    \item \textbf{Asynchroniczna weryfikacja:} w odpowiedzi zwracany jest tymczasowy token operacyjny (\textit{authenticationToken}) wraz z numerem referencyjnym, służący do odpytywania API przez klienta o status weryfikacji aż do uzyskania odpowiedzi o zakończeniu procesu. Jest to konieczne, ponieważ dla certyfikatów kwalifikowanych system weryfikuje w tle ich ważność u zewnętrznego wydawcy za pośrednictwem OCSP/CRL.
    
    \item \textbf{Odbiór JWT (\textit{Redeem}):} Gdy proces uwierzytelniania zakończy się sukcesem, klient jednorazowo wymienia token operacyjny na parę głównych tokenów dostępowych:
    \begin{itemize}
        \item \texttt{accessToken} -- krótkoterminowy, przekazywany w nagłówku \texttt{Authorization: Bearer} dla wszystkich autoryzowanych operacji,
        \item \texttt{refreshToken} -- długoterminowy (ważny do 7 dni), pozwalający na płynne odświeżanie sesji bez ponawiania operacji podpisywania.
    \end{itemize}
\end{enumerate}

\subsubsection{Cykl życia sesji}
\label{sec:ksef-cykl-zycia-sesji}

Po uwierzytelnieniu system udostępnia dwa rodzaje sesji do wysyłania faktur:

\begin{itemize}
    \item \textbf{Sesja interaktywna} (\texttt{POST /sessions/online}) -- synchroniczna operacja przeznaczona do wysyłki pojedynczych, zaszyfrowanych plików XML. Sesja ważna jest przez 12 godzin i można utrzymywać wiele takich sesji równolegle. W odpowiedzi na przesłanie faktury system natychmiast zwraca numer referencyjny. \cite{ksef-sesja-interaktywna}
    \item \textbf{Sesja wsadowa} (\texttt{POST /sessions/batch}) -- asynchroniczna operacja przeznaczona do masowej wysyłki plików ZIP. Paczka ZIP dzielona jest na części do 100 MB przed zaszyfrowaniem. \cite{ksef-sesja-wsadowa}
\end{itemize}

Cykl wysyłkowy kończy się zamknięciem sesji, po którym zwracane jest \textbf{Urzędowe Poświadczenie Odbioru (UPO)} -- dokument w formacie XML potwierdzający formalne przyjęcie faktury, stanowiący dowód dostarczenia dla celów podatkowych. \cite{ksef-sesja-sprawdzenie-stanu-i-pobranie-upo}

\subsubsection{Szyfrowanie i podpis}
\label{sec:ksef-szyfrowanie-i-podpis}

KSeF wymaga \textbf{szyfrowania wszystkich faktur}, zarówno w wysyłce masowej, jak i pojedynczej. Sama komunikacja odbywa się przez kanał zabezpieczony protokołem TLS, jednak dane biznesowe podlegają dodatkowemu szyfrowaniu asymetryczno-symetrycznemu (\textit{Envelope Encryption}). 

Każda faktura lub paczka faktur szyfrowana jest przez klienta lokalnie przy użyciu jednorazowego klucza w standardzie \texttt{AES-256-CBC} z dopełnieniem PKCS\#7. Wygenerowany klucz AES szyfrowany jest algorytmem \texttt{RSA-OAEP} (z użyciem algorytmu skrótu SHA-256 i funkcji maskującej MGF1) z wykorzystaniem klucza publicznego udostępnionego przez Ministerstwo Finansów. \cite{ksef-docs-przeglad,ksef-weryfikacja-faktury}

\subsubsection{Model odbioru faktur i synchronizacji danych}
\label{sec:ksef-odbieranie}

KSeF stanowi pasywne repozytorium, z którego klient samodzielnie pobiera dokumenty w modelu \textit{pull}. Cała komunikacja realizowana jest poprzez cykliczne odpytywanie API (\textit{polling}) -- system nie udostępnia natywnego mechanizmu \textit{push} (np.\ \textit{webhooków}). \cite{ksef-pobieranie-faktur}

\subsubsection*{Wyszukiwanie i pobieranie pojedynczej faktury} 

Do podstawowych operacji odbiorczych system udostępnia dwa punkty końcowe: \cite{ksef-pobieranie-faktur}

\begin{itemize}
    \item \texttt{GET /invoices/ksef/\{ksefNumber\}} -- zwraca treść XML konkretnej faktury identyfikowanej numerem KSeF nadanym przy wystawieniu,
    \item \texttt{POST /invoices/query/metadata} -- zwraca \textbf{stronicowaną listę metadanych} (m.in.\ numer KSeF, NIP-y stron, daty, kwoty) faktur spełniających kryteria filtra (\texttt{InvoiceQueryFilters}).
\end{itemize}

\subsubsection*{Eksport asynchroniczny} 

Pobieranie dużych zbiorów faktur (np.\ pełnej historii podmiotu) realizowane jest poprzez operacje asynchroniczne: \cite{ksef-pobieranie-faktur}

\begin{enumerate}
    \item \texttt{POST /invoices/exports} -- klient inicjuje operację, przekazując filtry oraz informacje o szyfrowaniu (klucz AES + IV), które posłużą do zabezpieczenia paczki wynikowej. W odpowiedzi otrzymuje numer referencyjny operacji.
    \item \texttt{GET /invoices/exports/\{referenceNumber\}} -- klient cyklicznie odpytuje endpoint o status operacji. W odpowiedzi zwracane są adresy URL zaszyfrowanych części archiwum ZIP.
\end{enumerate}

\subsubsection*{Przyrostowe pobieranie faktur} 

W przypadku utrzymywania lokalnej kopii repozytorium kluczowa jest \textit{ciągłość} pobierania danych. System adresuje ten problem poprzez mechanizm \textbf{High Water Mark (HWM)}, będącego punktem w czasie, do którego system gwarantuje kompletność danych. Wartość ta wyrażona jest datą trwałego zapisu faktury w repozytorium.

W modelu przyrostowym klient pobiera faktury w przylegających, nieprzecinających się oknach czasowych. Każda paczka eksportu zawiera jeden ze znaczników pozwalających wyznaczyć początek kolejnego okna: \cite{ksef-przyrostowe-pobieranie}

\begin{itemize}
    \item \texttt{PermanentStorageHwmDate} -- górna granica gwarancji kompletności; używana, gdy paczka nie została obcięta.
    \item \texttt{LastPermanentStorageDate} -- data ostatniej faktury w paczce, gdy paczka osiągnęła limit rozmiaru lub liczby faktur.
\end{itemize}
